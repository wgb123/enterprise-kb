# 项目介绍（简历版）

---

## Enterprise KB — 面向消费者的 AI 智能客服系统

> **关键词**：RAG、Hybrid Search、智能路由、Agent、GPU 加速
> **技术栈**：Python / FastAPI / Qdrant / BGE-M3 / rank-bm25 / LangChain / LlamaIndex

### 一、背景与场景

某 **互联网金融科技公司**（理财/交易平台，**千万级 C 端用户**），用户在使用产品时大量重复问答：

> "为什么我的提现还没到账？"
> "美股和港股的手续费怎么算？"
> "怎么开通融资融券权限？"

以前靠 50 人客服团队 + 工单系统 + FAQ 页面，平均响应时间 10 分钟，高峰期排队 1 小时。

### 二、知识分层设计

我把客服知识拆成 A/B 两类，各自入不同的库：

| 类别 | 存放内容 | 来源 | 特点 |
|------|---------|------|------|
| **Wiki（A 类编译式记忆）** | 产品 FAQ、业务规则、操作指南、费率表、开户流程、风控规则 | 产品/运营团队撰写，评审后入库 | 权威，人工维护，更新周期 ~1 月 |
| **HybridRAG（B 类检索式记忆）** | 监管文件（PDF）、合同条款（Word）、历史客诉 QA 记录（CSV）、客服培训手册 | 自动导入 | 海量（5000+ 文档），自动解析入库 |

### 三、一个完整的真实案例

```
用户在 APP 里输入：
"港股通的手续费怎么收？我昨天买了 1 万港币，扣了我 50 港币。"

步骤 1：SmartRouter 路由
  → "手续费" → 匹配 Wiki 信号（#fee, 费率）
  → "港股通" → 同时走 HybridRAG
  → 意图：HYBRID（两个引擎都查）

步骤 2：Wiki 命中（A 类）
  wiki/products/hk_stock_fee.md
  标题："港股通费用说明 v2.1"
  内容："港股通交易费率 0.25%，最低 50 港币。买入 1 万港币应收 25，但最低收费 50。"

步骤 3：HybridRAG 命中（B 类）
  Qdrant 搜到：港交所官方收费指引 PDF，确认 0.25% + 最低 50 港币
  BM25 搜到：历史客诉记录，"用户问过同样的 50 港币问题，客服回复已确认无多扣"

步骤 4：ContextFusion 融合
  → 综合答案："港股通费率是 0.25%（最低 50 港币），您买入 1 万港币应收 25，但最低收费 50 所以实际扣了 50，没有多扣。详细费率见..."
```

### 三、架构流程

```
                            ┌─ 路由信号 ─┐
                            │ Wiki信号    │
  "港股通手续费" ──SmartRouter──── 文档信号 ──→ HYBRID
                                          │
  ┌────────────────────────────────────────┘
  │                    │
  ▼                    ▼
WikiNavigator        HybridRetriever
(wiki/ 目录)           Qdrant(语义) + BM25(关键词)
  │                    │
  └──── ContextFusion ──┘
    Wiki 优先级 +0.3、按文本去重、来源标注
        │
     VLLMGenerator
    （DeepSeek / Qwen 兼容，一行 .env 切换）
        │
     答案 + 来源 [1][2]
```

### 四、核心设计亮点

**双模路由**：`ROUTER_TYPE=keyword`（关键词毫秒级，离线可用）/ `llamaindex`（LLM 语义路由），`BaseRouter` 抽象接口一键切换。

**三级排序防噪声**：Qdrant × BM25 并行 → RRF 融合（只看排名不看分数，消除量纲差异）→ BGE-Reranker 精排 TOP-5。效果：50 条候选压缩到 5 条，低质量结果不过 LLM。

**BGE-M3 GPU 加速**：RTX 4060 本地推理，批处理 32 篇/次 ~1,000 篇/秒；支持 Dense + Sparse + ColBERT 三模式；显存不足自动降级 CPU。

**Agent 工具链**：面向**消费者**——用户在 APP 里输入问题，Agent 自动检索业务知识 + 查询订单状态 + 做费用计算 + 返回带来源的答案。替代 50 人客服团队解决 80% 的重复问题，人工客服只处理复杂投诉。**数据完全私有部署**，用户金融信息不经过第三方 API。

**断点续传**：所有 processed IDs 完整保存，BM25 / 向量独立 checkpoint，Qdrant 自动快照。

## Dota 2 AI Agent（Rust 全栈 + LLM 回测系统）

> **关键词**：Rust、Axum、Tokio、LLM Agent、多数据源聚合、SSE、Prompt 回测、Polymarket
> **技术栈**：Rust / Axum / Tokio / GraphQL / SSE / DeepSeek / Qwen

### 一、要解决的问题

Dota 2 比赛分析需要同时参考三个数据源，人工操作 15 分钟且易遗漏。目标是：**一句话输入阵容，AI 自动拉取多源数据、多步推理并输出结构化的分析报告**，同时将结论与预测市场赔率对比验证。

### 二、系统架构

```
┌─ 前端 / API ─────────────────────────────────┐
│  输入：阵容 / 比赛 ID          输出：SSE 实时进度  │
└────────────────────┬──────────────────────────┘
                     │ HTTP / SSE
                     ▼
┌─ 服务层：Rust Axum ─────────────────────────┐
│  ┌─ Data Ingestion Layer ───────────────┐    │
│  │  STRATZ (GraphQL)                    │    │  ← tokio::join!
│  │  OpenDota (REST, 429 退避)           │    │  ← TTL 缓存
│  │  Liquipedia (HTML 解析)              │    │  ← 去重合并
│  └───────────────────┬──────────────────┘    │
│                      │                        │
│  ┌───────────────────▼──────────────────┐    │
│  │  Context Assembly Layer              │    │
│  │  原始 JSON/HTML → LLM 可读的结构文本  │    │
│  └───────────────────┬──────────────────┘    │
│                      │                        │
│  ┌───────────────────▼──────────────────┐    │
│  │  LLM Provider 抽象层                 │    │
│  │  DeepSeek / Qwen / GPT 热切换        │    │
│  └───────────────────┬──────────────────┘    │
│                      │                        │
│  ┌───────────────────▼──────────────────┐    │
│  │  Agent 决策流水线 (5 步)             │    │
│  └───────────────────┬──────────────────┘    │
└──────────────────────┬───────────────────────┘
                       │
┌──────────────────────▼───────────────────────┐
│  存储层：TTL Cache + Prompt 归档 + 回测结果   │
└──────────────────────────────────────────────┘
```

### 三、核心组件

**3.1 数据采集层**：STRATZ（GraphQL，支持 fragment 复用）、OpenDota（REST，429 指数退避）、Liquipedia（HTML 解析），三源 `tokio::join!` 并发，TTL 缓存（英雄 1h / 选手 24h）。

**3.2 Context Assembly**：原始 JSON / HTML → 统一结构化文本模板，每源打上 `{source, timestamp, confidence}` 元数据（LLM 读不了嵌套 GraphQL 响应，这一层解决"格式对齐"）。

**3.3 LLM Provider 抽象**：`LlmProvider` trait + 三种实现（DeepSeek 低成本 / Qwen 合规部署 / GPT baseline），热切换保留上下文。

**3.4 Agent 决策流水线**（5 步，每步 SSE 推送进度）：
```
① 阵容分析 → ② 选手状态 → ③ 版本影响 → ④ 综合预测 → ⑤ Polymarket 赔率对比
```

**3.5 Prompt 回测体系**：每个 prompt 版本对 20 个历史比赛跑对比，3 个量化指标（准确率 / 完整性 / 幻觉率），数据驱动优化。

**3.6 差异化价值**：Polymarket 预测市场集成（LLM 结论 vs 真实赔率差异分析）；Rust 选型理由（GSI 实时流零开销 + tokio 真并行 + 编译期 API 兼容性保障）。

---

### 四、两项目共通设计思想

```
→ 都在解决「LLM 在真实场景下的可靠性」
→ Enterprise KB：多源检索 + 交叉验证 + Abstain → 信息准确
→ Dota 2 Agent：多步推理 + 工具调用 + 回测量化 → 任务完成
→ 共同前提：通过分层抽象（Router / Provider / Agent Loop）解耦
```
