> 最后更新：2026-06-20

## 1. 项目中遇到的最大困难是什么？怎么解决的？

**问题背景**：中文维基百科全量导入（3.3GB bz2 XML，解压后约 60GB+），需要拆成 85,000+ 篇文章，每篇清洗 Wiki 标记（`[[链接]]`、`{{模板}}`、`<ref>` 等 10 种正则），再分块入库到 BM25 和 Qdrant。

**困难有三个**：

① **内存爆炸**：一次性加载整个 XML 解压流会吃 60GB。早期用 `etree.parse()` 全量加载，OOM 死机。

② **断点续传**：导入 6 小时，中途电脑休眠 / 网络断了 / OOM 挂了 → 进度全部丢失，得重来。

③ **BGE-M3 模型下载**：30 个文件共 5.8GB，国内网络不稳定。HuggingFace 下载频繁中断，`snapshot_download` 的断点续传被 SOCKS 代理干扰（httpx 把 HTTP 代理当 SOCKS 用，报 `socksio` 缺失）。

**解决方案**：

- **流式解析**：改用 `lxml.etree.iterparse()`，逐 `<page>` 元素流式读取 → 处理 → `elem.clear()` 释放内存。3.3GB 文件处理全程内存 < 500MB。
- **双 checkpoint 断点保护**：每 500 篇做一次 `_flush_buffers()`（写入 Qdrant + 持久化 BM25）+ `_save_checkpoint()`（记录已处理的 page ID）。重启时跳过已处理文章。导入 85,000 篇一次完成，中间中断 3 次都无损失。
- **网络问题**：发现 `ALL_PROXY=socks5://...` 是根因。修复 `.bashrc` 只设 `HTTP_PROXY`，不设 `all_proxy`。然后通过预编译 wheel（`lxml>=6.0`）绕开 Python 3.14 的编译环境问题。

**面试升华**：这个问题的本质是 **「如何在不可靠环境中处理大数据流的可靠性」**。不只是导入——日志采集、消息队列、数据迁移都会遇到类似问题。我们的方案（流式处理 + 断点保护 + 环境隔离）可以复用到任何大数据管道。

## 2. 从 0 到 1 构建一个智能体的思路是什么？

**五个步骤，层层递进**：

### 第一步：定义边界（最重要的 20%）

不要一开始就想"通用智能体"。先回答三个问题：

```
① 它能做什么？不能做什么？  → 路由分发：Wiki/RAG/通用
② 用户输入长什么样？        → 单轮查询 vs 多轮对话
③ 输出质量怎么衡量？        → 答案准确性 + 来源可追溯
```

本项目就是典型——"企业知识库助手"这个边界很清晰，不需要做开放式聊天、代码生成等。

### 第二步：设计架构（松耦合）

```
用户请求 → [路由层] → [检索层] → [融合层] → [生成层] → 回答
                        ↓
                   [工具层] ← [Agent 层]
```

每层独立可测试、可替换、可降级。本例中：
- 路由层可以切 `SmartRouter`（手写）↔ `LlamaIndexRouter`（LLM）
- 检索层可以切 `Qdrant（向量）` ↔ `BM25（关键词）` ↔ 两者融合
- 生成层可以切 `VLLMGenerator` ↔ `LcGenerator`（LangChain）

### 第三步：实现核心闭环

先不要管边缘情况，让最小闭环跑起来：

```
用户说"帮我查一下Python的安装" 
  → 路由判断走文档检索
  → Qdrant 向量搜索返回相关片段
  → VLLM 根据片段生成回答
  → 返回"Python安装步骤：..."
```

这条线通了，整个系统的骨架就有了。

### 第四步：加鲁棒性（80% 的工作量在这个阶段）

```
- 检索不到 → 降级回复（"未找到相关信息，请换个关键词"）
- LLM API 超时 → 直接返回最相关的原文片段
- Qdrant 挂了 → 只用 BM25 检索
- 模型加载失败 → 自动降级到 CPU
- 两路冲突 → Wiki 权威覆盖 HybridRAG
```

### 第五步：可观测性

```
- 每条查询记录：intent + 引擎选择 + 耗时
- 每批次导入记录：进度 + 速度 + 错误数
- 异常时自动日志告警
```

**面试话术**：不要背这五步，要说 **"我的核心原则是『先跑通再加固』——先让 20% 的核心场景能工作，再花 80% 的精力处理边界情况和降级"**。

## 3. RAG 的完整流程

### 一句话

RAG = **Retrieve**（从知识库中找到相关片段）+ **Augment**（把片段拼到 prompt 里）+ **Generate**（让 LLM 基于片段生成答案）。

### 本项目的 RAG 流程（六步）

```
[ 用户查询 ]
     │
     ▼
① 路由分发（SmartRouter）
     │   ├── Wiki 关键词 → WikiNavigator（本地 Markdown）
     │   ├── 文档关键词 → HybridRetriever（向量+BM25）
     │   └── 通用/混合 → 两路并行
     │
     ▼
② 向量检索（Qdrant）
     │   query → BGE-M3 embed → 余弦相似度搜索 → top_k 候选
     │
     ▼
③ 关键词检索（BM25）
     │   query 分词 → BM25Okapi 打分 → top_k 候选
     │
     ▼
④ RRF 融合排序
     │   score = Σ 1 / (k + rank)    其中 k=60
     │   合并两路结果，去重
     │
     ▼
⑤ BGE-Reranker 重排序
     │   交叉编码器（Cross-encoder）逐对打分
     │   对 top_10 重新排序，比向量余弦距离更准
     │
     ▼
⑥ VLLM（LLM）生成答案
     │   prompt = 系统指令 + 检索片段 + 用户问题
     │   LLM 基于上下文生成，引用 [来源]
     │
     ▼
[ 最终答案 ]
```

### 关键决策点

```python
# 为什么向量+BM25两路？ → 语义相似 + 精确匹配互补
# 为什么 RRF 而不是加权平均？ → 不需要调权，鲁棒
# 为什么只用摘要向量化？ → 150,000 篇文章，全文向量化太贵
# 为什么还要加 Reranker？ → 粗排（向量）→ 精排（Cross-encoder）两阶段
```

### 架构变体

```
普通 RAG：  检索 → LLM 生成（本项目就是这种）
Agent RAG： 检索 → Agent 思考 → 决定是否再检索 → LLM 生成
Graph RAG： 检索 → 知识图谱推理 → LLM 生成（本项目预留 C 类）
```

## 4. MCP 是什么？

**MCP（Model Context Protocol）**——Anthropic 推出的**大模型与外部工具的标准化协议**。

### 一句话解释

```
MCP = AI 界的 USB 协议

USB：键盘、鼠标、U盘都插同一个接口，即插即用
MCP：数据库、API、文件系统都连同一个协议，即插即用
```

### 传统方式 vs MCP

```python
# 传统：每个工具自己写一套连接逻辑
def query_database(sql):
    import psycopg2
    conn = psycopg2.connect(...)  # 每个工具自己管连接

def search_web(q):
    import requests
    resp = requests.get(...)      # 另一个工具自己管连接

# MCP：所有工具通过统一协议连接
# 工具方：实现 MCP Server（提供 tools + resources）
# 调用方：通过 MCP Client 连接，传入 {tool, params}
# 标准格式：JSON-RPC 2.0
```

### MCP 三要素

```
MCP Host       — 发起请求的程序（如 Claude Desktop、我们的 Agent）
MCP Client     — 与 Server 建立 1:1 连接
MCP Server     — 暴露工具（tools）、资源（resources）、提示（prompts）
```

### 本项目的 MCP 视角

如果把本项目封装成 MCP Server，大概长这样：

```
工具列表：
  wiki_search(query)      → 搜索 A 类编译式记忆
  rag_search(query)       → 搜索 B 类向量知识库
  ingest(file_path)       → 文档入库
  calculator(expr)        → 安全计算器

资源列表：
  enterprise_kb://wiki/   → Wiki 知识库
  enterprise_kb://docs/   → 导入的文档
```

**面试话术**：MCP 的价值不在于技术有多新，而在于**「标准化」**——就像 HTTP 标准化了 Web 通信，MCP 标准化了 AI 与工具的通信。谁先支持、谁生态做得好，谁就有先发优势。

## 5. 向量算法了解吗？

### 核心概念

向量算法本质上解决一个问题：**如何衡量两个事物的相似度**。

```python
"猫"  → [0.2, 0.5, 0.1, ...]   # 嵌入向量
"狗"  → [0.3, 0.6, 0.2, ...]   # 语义相近 → 向量相近
"汽车" → [0.9, 0.1, 0.8, ...]  # 语义不同 → 向量不同
```

### 三种相似度度量

| 算法 | 公式 | 特点 | 本项目用 |
|------|------|------|---------|
| **余弦相似度** | `cos(A,B) = A·B / \|A\|\|B\|` | 只看方向，不看长度。文本语义最常用。| ✅ Qdrant 默认 |
| **点积** | `A·B = Σ aᵢbᵢ` | 受向量长度影响。需要归一化后才公平。 | ❌ |
| **欧几里得距离** | `\|\|A-B\|\| = sqrt(Σ (aᵢ-bᵢ)²)` | 数值敏感。图像检索常用。 | ❌ |

### 向量搜索算法

```text
暴力搜索：O(n)  每个向量都比一次 → 100 万条就 100 万次计算 → 太慢

近似搜索（ANN）：O(log n) 用索引加速 → 本项目的 Qdrant 核心

Qdrant 用的 HNSW（Hierarchical Navigable Small World）：
  原理：多层图结构
    Layer 3:    ●────●────●      (长连接，粗搜索)
    Layer 2:    ●──●──●──●      (中间层)
    Layer 1:    ●─●─●─●─●─●    (短连接，精搜索)
    
  搜索过程：
    1. 从顶层（Layer 3）找一个入口点
    2. 沿连接找到更近的节点
    3. 下到下一层继续找
    4. 到最底层（Layer 1）返回最近邻居
  
  效果：10 万条数据，比暴力搜索快 100 倍，准确率 99%+
```

### BGE-M3 的特点（本项目用的嵌入模型）

```
多语言      ✅ 中英文混合搜索
多粒度      ✅ 支持 token / sentence / document 三种粒度
多向量      ✅ 可以同时输出稠密向量（dense）+ 稀疏向量（sparse/lexical）
1024 维     ✅ 比常见的 768 维承载更多语义信息
```

**面试话术**：不要只背概念，要说 **"我在项目中做了什么"**——"我们对比了 `FlagEmbedding` 和 API 两种嵌入方式，最终选 BGE-M3 因为它本地可部署、支持中文、1024 维精度够。向量检索用 Qdrant HNSW，配合 BGE-Reranker 做精排，两阶段架构在精度和速度之间取得平衡。"

## 6. 如何解决大模型幻觉问题？

### 本项目的三层防幻觉体系

```text
① 检索层 — 不给 LLM 编造的机会
   知识库先行，LLM 只能基于检索到的上下文回答
   prompt 中强制规定："只使用上下文中的信息回答，不要编造事实"

② 融合层 — 权威覆盖
   Wiki 知识（A 类）优先级最高，内容冲突时覆盖 HybridRAG（B 类）
   如果检索不到 → 直接说"根据现有知识库无法回答"，不编造

③ 生成层 — 来源可追溯
   回答必须标注 [来源: 文档名称]
   用户可以看到答案来自哪篇文档
```

### prompt 层面

```python
SYSTEM_PROMPT = """
你是一个企业知识库助手。请根据提供的上下文信息回答用户问题。

规则：
1. 只使用上下文中的信息回答，不要编造事实
2. 如果上下文不足以回答问题，请说"根据现有知识库无法回答这个问题"
3. 在回答中引用来源，使用 [来源: 文档名称] 格式
4. 如果多个来源的信息有冲突，优先采信 Wiki 知识库的内容
"""
```

### 两种降级模式

```python
# ① LLM 正常 → 输出 LLM 生成的答案
# ② LLM 失败（API 超时 / 报错）→ 直接返回最相关的原文片段
# ③ LLM 没检索到内容 → 返回"根据现有知识库无法回答"
```

### 面试话术

"幻觉的本质原因是**LLM 没有"不知道"的概念**——它被训练成永远要输出一堆 token。解决方案不是让 LLM 变得更"诚实"，而是**从架构上不让它有编造的机会**：先检索，再生成，只回答检索到的内容。"

## 7. 如何保证大模型格式输出？

### 三种方法（从弱到强）

**方法 1：prompt 约束（最弱）**

```python
prompt = "请严格输出 JSON 格式：{\"answer\": \"...\", \"sources\": [...]}"
```

⚠️ 缺点：LLM 不一定听话，偶尔会加额外文字。

**方法 2：输出解析器（中等）**

```python
from langchain_core.output_parsers import PydanticOutputParser

class Answer(BaseModel):
    answer: str
    sources: list[str]
    confidence: float

parser = PydanticOutputParser(pydantic_object=Answer)
chain = prompt | llm | parser  # ← 自动解析+校验
# LLM 输出 -> parser 校验 -> 如果格式不对，自动重试
```

**方法 3：结构化输出 API（最强）**

```python
# OpenAI / DeepSeek 都支持 response_format 参数
response = client.chat.completions.create(
    model="deepseek-chat",
    messages=[...],
    response_format={"type": "json_object"},  # ← 强制 JSON
)
```

本项目的方案：**方法 1（prompt）+ 方法 2（异常捕获）组合**——prompt 要求格式化，但如果格式不对，不会让 LLM 重试而是直接返回原文降级。因为我们的场景不需要严格结构化（最终是给人看），格式错了不影响使用。

如果场景要求**严格的结构化输出**（如调用 API、数据库写入），就必须用方法 3。

## 8. 国内外大模型区别与看法

### 核心差异

| 维度 | 国内（DeepSeek / Qwen / GLM） | 国外（OpenAI / Claude / Gemini） |
|------|------------------------------|----------------------------------|
| **中文能力** | 强。DeepSeek 的中文理解接近 native | 中等。GPT-4 中文还不错，但成语/俗语有时跑偏 |
| **价格** | 极低。DeepSeek API 约 $0.14/M 输入 | 贵。GPT-4o 约 $2.5/M 输入（~18x） |
| **开放程度** | 高。DeepSeek 开源权重，可本地部署 | 低。OpenAI 闭源，只能用 API |
| **长上下文** | DeepSeek 1M token | GPT-4o 128K，Claude 200K |
| **生态** | 还在建设中，工具少 | 成熟。LangChain / LlamaIndex 优先适配 |
| **合规** | 数据不出境，适合企业内网 | 数据出境，合规门槛高 |

### 本项目的选择

```
核心原则：API 兼容性优先

本项目用的是 OpenAI 兼容接口（/v1/chat/completions）
→ 可以无缝切换任何兼容的模型：

# .env 改一行就行
VLLM_API_URL=https://api.deepseek.com/v1    # DeepSeek
VLLM_API_URL=https://api.openai.com/v1      # OpenAI
VLLM_API_URL=http://localhost:8000/v1        # 本地 vLLM 部署
```

### 面试看法

说三点，显得有深度：

1. **「地缘政治是现实，不是幻想」**——国内企业做 To B 产品，必须考虑制裁风险。DeepSeek 开源+低价+中文强，是目前国内企业的第一选择。但也要保持 API 兼容性，哪天政策变了可以无缝切到 OpenAI。

2. **「开源 vs 闭源最终会走向两极分化」**——闭源模型（GPT-4o / Claude）靠 huge scale 碾压，适合复杂推理（写代码、数学证明）；开源模型（DeepSeek / Qwen / LLaMA）靠低成本+可定制，适合垂直场景（企业内部知识库、客服）。这是**两条赛道**，不是谁取代谁。

3. **「MCP 统一协议是一个转折点」**——以前选模型=锁死生态（选了 OpenAI 就不方便切 DeepSeek）。MCP 标准化工具接口后，底层模型可以随时换——API 层是 OpenAI，本地则用 DeepSeek，MCP 层不用改。**选框架比选模型更关键。**

## 1. 项目中遇到的最大困难是什么？怎么解决的？

**问题背景**：中文维基百科全量导入（3.3GB bz2 XML，解压后约 60GB+），需要拆成 85,000+ 篇文章，每篇清洗 Wiki 标记（`[[链接]]`、`{{模板}}`、`<ref>` 等 10 种正则），再分块入库到 BM25 和 Qdrant。

**困难有三个**：

① **内存爆炸**：一次性加载整个 XML 解压流会吃 60GB。早期用 `etree.parse()` 全量加载，OOM 死机。

② **断点续传**：导入 6 小时，中途电脑休眠 / 网络断了 / OOM 挂了 → 进度全部丢失，得重来。

③ **BGE-M3 模型下载**：30 个文件共 5.8GB，国内网络不稳定。HuggingFace 下载频繁中断，`snapshot_download` 的断点续传被 SOCKS 代理干扰（httpx 把 HTTP 代理当 SOCKS 用，报 `socksio` 缺失）。

**解决方案**：

- **流式解析**：改用 `lxml.etree.iterparse()`，逐 `<page>` 元素流式读取 → 处理 → `elem.clear()` 释放内存。3.3GB 文件处理全程内存 < 500MB。
- **双 checkpoint 断点保护**：每 500 篇做一次 `_flush_buffers()`（写入 Qdrant + 持久化 BM25）+ `_save_checkpoint()`（记录已处理的 page ID）。重启时跳过已处理文章。导入 85,000 篇一次完成，中间中断 3 次都无损失。
- **网络问题**：发现 `ALL_PROXY=socks5://...` 是根因。修复 `.bashrc` 只设 `HTTP_PROXY`，不设 `all_proxy`。然后通过预编译 wheel（`lxml>=6.0`）绕开 Python 3.14 的编译环境问题。

**面试升华**：这个问题的本质是 **「如何在不可靠环境中处理大数据流的可靠性」**。不只是导入——日志采集、消息队列、数据迁移都会遇到类似问题。我们的方案（流式处理 + 断点保护 + 环境隔离）可以复用到任何大数据管道。

## 2. 从 0 到 1 构建一个智能体的思路是什么？

**五个步骤，层层递进**：

### 第一步：定义边界（最重要的 20%）

不要一开始就想"通用智能体"。先回答三个问题：

```
① 它能做什么？不能做什么？  → 路由分发：Wiki/RAG/通用
② 用户输入长什么样？        → 单轮查询 vs 多轮对话
③ 输出质量怎么衡量？        → 答案准确性 + 来源可追溯
```

### 第二步：设计架构（松耦合）

```
用户请求 → [路由层] → [检索层] → [融合层] → [生成层] → 回答
                        ↓
                   [工具层] ← [Agent 层]
```

每层独立可测试、可替换、可降级。例如路由层可以切 `SmartRouter` ↔ `LlamaIndexRouter`。

### 第三步：实现核心闭环

先不要管边缘情况，让最小闭环跑起来：

```
用户说"帮我查一下Python的安装" 
  → 路由判断走文档检索
  → Qdrant 向量搜索返回相关片段
  → VLLM 根据片段生成回答
  → 返回"Python安装步骤：..."
```

### 第四步：加鲁棒性（80% 的工作量在这个阶段）

```
- 检索不到 → 降级回复
- LLM API 超时 → 直接返回最相关的原文片段
- Qdrant 挂了 → 只用 BM25 检索
- 模型加载失败 → 自动降级到 CPU
- 两路冲突 → Wiki 权威覆盖 HybridRAG
```

### 第五步：可观测性

```
- 每条查询记录：intent + 引擎选择 + 耗时
- 每批次导入记录：进度 + 速度 + 错误数
```

**面试话术**：不说这五步，而是说 **"核心原则是『先跑通再加固』——先让 20% 的核心场景工作，再花 80% 的精力处理边界情况和降级"**。

## 3. RAG 的完整流程

### 本项目的 RAG 流程（六步）

```
① 路由分发（SmartRouter）→ 决定 Wiki / RAG / 两路并行
② 向量检索（Qdrant + BGE-M3）→ 语义相似度搜索
③ 关键词检索（BM25Okapi）→ 精确匹配
④ RRF 融合排序 → 合并两路，去重
⑤ BGE-Reranker 重排序 → 交叉编码器精排
⑥ VLLM 生成答案 → 基于上下文 + 引用来源
```

### 关键决策

```python
# 为什么向量+BM25两路？ → 语义相似 + 精确匹配互补
# 为什么 RRF 而不是加权平均？ → 不需要调权，鲁棒
# 为什么只用摘要向量化？ → 150,000 篇文章，全文向量化太贵
# 为什么还要加 Reranker？ → 粗排（向量）→ 精排（Cross-encoder）两阶段
```

## 4. MCP 是什么？

**MCP（Model Context Protocol）**——大模型与外部工具的标准化协议。

```
MCP = AI 界的 USB 协议
USB：键盘、鼠标、U盘都插同一个接口，即插即用
MCP：数据库、API、文件系统都连同一个协议，即插即用
```

### 三要素

```
MCP Host   — 发起请求的程序（如 Claude Desktop、我们的 Agent）
MCP Client — 与 Server 建立 1:1 连接
MCP Server — 暴露工具（tools）、资源（resources）、提示（prompts）
```

**面试话术**：MCP 的价值不在于技术有多新，而在于**「标准化」**——就像 HTTP 标准化了 Web 通信，MCP 标准化了 AI 与工具的通信。

## 5. 向量算法了解吗？

### 三种相似度度量

| 算法 | 公式 | 特点 |
|------|------|------|
| **余弦相似度** | `cos(A,B) = A·B / \|A\|\|B\|` | 只看方向，不看长度。文本语义最常用 |
| **点积** | `A·B = Σ aᵢbᵢ` | 受向量长度影响 |
| **欧几里得距离** | `\|\|A-B\|\| = sqrt(Σ (aᵢ-bᵢ)²)` | 数值敏感 |

### HNSW 算法（本项目 Qdrant 用的）

```
多层图结构，搜索时从顶层粗搜 → 逐层下降 → 底层精搜
效果：10 万条数据，比暴力搜索快 100 倍，准确率 99%+
```

### BGE-M3 特点

```
多语言      ✅ 中英文混合搜索
多粒度      ✅ token / sentence / document 三种粒度
1024 维     ✅ 比常见的 768 维承载更多语义信息
```

## 6. 如何解决大模型幻觉问题？

### 本项目的三层防幻觉体系

```
① 检索层 — 不给 LLM 编造的机会
   知识库先行，prompt 强制："只使用上下文信息回答，不要编造事实"

② 融合层 — 权威覆盖
   Wiki 内容冲突时覆盖 HybridRAG
   检索不到 → 说"根据现有知识库无法回答"

③ 生成层 — 来源可追溯
   回答必须标注 [来源: 文档名称]
```

## 7. 如何保证大模型格式输出？

### 三种方法（从弱到强）

| 方法 | 实现 | 可靠性 |
|------|------|--------|
| Prompt 约束 | `"请严格输出JSON"` | ⚠️ 不可靠 |
| 输出解析器 | `PydanticOutputParser` 自动校验+重试 | ✅ 较可靠 |
| 结构化输出 API | `response_format={"type": "json_object"}` | ✅✅ 最强 |

## 8. 国内外大模型区别与看法

### 核心差异

| 维度 | 国内（DeepSeek / Qwen） | 国外（GPT-4o / Claude） |
|------|------------------------|-------------------------|
| 中文能力 | 强 | 中等 |
| 价格 | 极低（$0.14/M） | 贵（$2.5/M，~18x） |
| 开放程度 | 开源，可本地部署 | 闭源，只能用 API |
| 长上下文 | DeepSeek 1M token | GPT-4o 128K |
| 合规 | 数据不出境 | 数据出境 |

### 面试三句话

1. **地缘政治是现实**——国内企业必须考虑制裁风险，DeepSeek 是目前第一选择。但要保持 API 兼容性，随时能切。
2. **开源 vs 闭源**——两条赛道：闭源靠 huge scale 碾压，适合复杂推理；开源靠低成本+可定制，适合垂直场景。
3. **选框架比选模型更关键**——MCP 标准化后，底层模型可随时换。
