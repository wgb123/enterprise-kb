# 面试准备：三道必考题话术

---

## 第一题：「给我讲一下你做的 Agent 系统是怎么工作的」

这是**最最最重要的一道题**，决定 80% 的第一印象。面试官不是要听代码细节，而是在判断你有没有「从 0 到 1 的架构视角」。

### ❌ 错误示范

> "我用了 LangChain 和 Qdrant，用户输入查询后，先路由到不同的引擎，然后检索，然后生成回答，还接入了 DeepSeek API……"

**问题**：平铺直叙、全是技术名词堆砌、没有"为什么"。

### ✅ 正确示范（5 分钟版本）

**第 1 分钟：一句话定位**

> "我参与设计并实现了一个企业级知识库 Agent 系统。核心思路是**解决了企业知识管理的三个层级**："
>
> "- **A 层**：编译式记忆 —— 公司内部架构文档、开发规范这类经过人工审核的高质量内容，存为本地 Markdown，路由到独立的 Wiki 引擎，**优先级最高，作为单一真相来源**"
> "- **B 层**：HybridRAG 检索式记忆 —— 海量文档（PDF/Word），通过文本分块、向量嵌入和 BM25 关键词建立索引，走混合检索管道，**覆盖广度**"
> "- **C 层**：预留的 GraphRAG 接口"

**效果**：第一句话就展示了你对「分层架构」的理解，不是只会搭 RAG。

---

**第 2 分钟：Agent Loop**

> "在这三层记忆之上，我实现了一个**基于 Function Calling 的 Agent 循环**。核心流程是 **Think → Act → Observe**："
>
> "1. **Think**：用户提了一个问题，LLM 先思考需要哪些信息"
> "2. **Act**：LLM 决定调用工具 —— 比如搜索 Wiki、检索 RAG、或者算一个数字"
> "3. **Observe**：工具返回结果，LLM 观察结果，决定是需要更多信息还是可以给出最终答案"
>
> "这套循环是我**手写的，没有依赖 LangChain**。核心只有三个组件：一个 ToolRegistry 管理工具注册、一个 ToolExecutor 执行工具调用、一个 Memory 维护对话历史。每一步都有 max_steps 保护，防止死循环。"

**效果**：展示你对 Agent 本质的理解，而不是「用框架搭了一个」。

---

**第 3 分钟：RAG 检索链路**

> "在 B 层的混合检索管道的设计上，是这样的链路："
>
> "查询 → BGE-M3 嵌入向量 → Qdrant 向量检索 → BM25 关键词检索 → RRF 融合排序 → BGE-Reranker 重排序 → 生成"
>
> "这里做了几个关键决策："
> "- **向量 + 关键词双通道**：单靠向量会丢失专有名词的匹配，加了 BM25 后准确率提升了约 30%"
> "- **RRF 融合不用 weights**：不需要调参，k=60 的经验值在不同数据集上表现稳定"
> "- **Reranker 是瓶颈**：交叉编码器要算 query+doc 对，N=50 时就是 50 次推理，所以只在最后一步做精排"

**效果**：展示你**做了权衡决策**，不是照着教程搭的。

---

**第 4 分钟：LLM 回测体系**

> "在 Dota 2 项目中，我做了一套 **Prompt 回测体系** —— 每次 LLM 分析请求的完整 prompt + response 都会被归档，后续可以回放对比不同 Provider、不同 prompt 版本的效果差异。这在企业落地中是必须的，因为 LLM 输出不稳定，需要能追踪和评估。"

**效果**：这是大多数人没有的差异化亮点。

---

**第 5 分钟：收尾总结**

> "这整个系统的设计原则是：**能手写的部分不依赖框架**。用了框架后用不了多久就会遇到框架的边界，然后花大量时间 debug 框架本身。所以核心的 Agent Loop 和 RAG 管道是自己实现的，框架只用来做对比参考。"
>
> "当然，如果团队已经深度使用了某套框架，我也能快速上手 —— 我已经写了 LangChain 和 Dify 的对比分析，理解它们的核心抽象。"

**效果**：展示务实态度，同时不排斥团队技术栈。

## 第二题：「遇到过什么坑，怎么解决的」

这道题考的是**解决问题的能力**，不是考你犯了多少错。

### 题库（选 2-3 个准备）

**坑 1：LLM 总是重复调用工具**

> **现象**：Agent 调了一次工具，结果不太满意，接着又调一次，第三次才给出答案，浪费了 2 步。
>
> **解决**：在 system prompt 里加了一条规则「如果你已经获取了足够信息，请直接给出答案，不要重复调工具」。同时加了 `max_steps=10` 的硬保护。

**坑 2：Qdrant 写入太慢**

> **现象**：维基百科导入时，单篇逐条 upsert，速度只有 50 篇/秒。
>
> **解决**：改成批量写入，`batch_size=200`，速度提升到 800 篇/秒。同时发现了 `wait=True` 是同步等待写入完成，改为 `wait=False` 后写入是异步的。

**坑 3：BM25 全量重建**

> **现象**：每次新加文档都要重建整个 BM25 索引，N=1万 时花了 30 秒。
>
> **解决**：目前还没做增量更新，但明确了这是一个架构上的 trade-off —— 全量简单可靠但慢，增量复杂但快。在面试里说出来"我知道这个问题，下一版计划做增量更新"比假装没问题强。

**坑 4：Dota 2 数据分析是英文，LLM 用中文回答时英文数据丢失**

> **现象**：英雄名、技能名在英文原文是准确的专业术语，但 LLM 翻译成中文后容易歧义。
>
> **解决**：prompt 里要求「专用名词保留英文原文，分析用中文」。这是一个典型的 **Prompt 工程**问题。

### 通用回答框架

> "我遇到的一个典型问题是 XX。当时的表象是 XX。
> **排查过程**：我先看了 XX 日志，又测试了 XX 场景，排除了 XX 的原因，最后定位到 XX。
> **解决方式**：我做了 XX 改进，改完之后速度从 X 提升到了 Y。
> **我的反思**：如果再让我做一次，我会在一开始就 XX，而不是等出了问题再修。"

## 第三题：「让你设计一个 XX 场景的 Agent，你打算怎么做」

这道题考的是**架构设计思维**，不是已经有标准答案的东西。

### 回答框架

> "我会分三步设计："

> **第一步：梳理需求边界**
> "先问清楚：用户是谁？输入是什么？期望输出是什么？数据源在哪里？对实时性有要求吗？"
> **（展示你不会上来就写代码，而是先理解场景）**

> **第二步：确定架构模式**
> "这个场景适合用哪种 Agent 模式：
> - 如果只是简单的问答 → 单轮 RAG 就够，不需要 Agent Loop
> - 如果需要多步检索再汇总 → 用 ReAct 模式的 Agent Loop
> - 如果需要和外部系统交互（查数据库、调用 API） → Function Calling Agent"
> **（展示你能根据场景做选择，而不是一招鲜）**

> **第三步：落地方案**
> "数据层用 Qdrant 做向量存储 + MySQL 做元数据管理。Agent 层我自己实现，不依赖框架。部署用 Docker Compose，API 用 FastAPI。监控加一个日志审计——每次 Agent 决策的轨迹都记录下来，方便回放排查。"
> **（展示你考虑的不仅仅是代码，还有监控和运维）**

## 面试避坑清单

| 坑 | 应该怎么说 |
|---|---|
| "我用 LangChain 搭了一个 RAG" | "我手写了核心流程，同时对比了 LangChain 的实现，理解了它的设计取舍" |
| "这个项目用了 OpenAI / DeepSeek" | "项目接入了多个 LLM Provider，通过抽象层可以随时切换" |
| "Qdrant 是最好的向量数据库" | "Qdrant 适合我们这个场景是因为…… 但如果场景变成 XX，也可能会考虑 Milvus" |
| "我不知道，没遇到过" | "这个场景我暂时没碰到，但我可以分享一下我理解的思路……" |
| 面试官质疑技术选型 | 先认同再解释：「你说得对，XX 方式确实有优势，我当时的考虑是 XX，因为我们的场景更关注 XX」 |

## 一句话自我介绍模板

> "我做过两个 Agent 项目。一个是**企业知识库 Agent**，Python 技术栈，手写了 RAG 管道和 Agent Loop，不依赖 LangChain；另一个是 **Dota 2 电竞分析 Agent**，Rust 技术栈，集成了多数据源和 LLM 回测体系。两个都是从 0 到 1 的落地项目。"

**40 个字，三个关键信息**：做过什么、技术栈、落地了。
