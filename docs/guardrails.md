# 破坏性操作守则（2026-06-20 立）

## 教训
查看向量导入进度时，未报告异常就直接删除了 Qdrant collection，
导致数据丢失需要重导。用户明确批评了「不询问直接删库」。

## 硬规则
任何破坏性操作前，必须先报告现状 + 提方案 + 等用户确认：

1. 这是破坏性操作吗？（DELETE / DROP / 清空 / 覆盖 / 重置 / 重命名 / 移除）
2. 我向用户完整报告现状了吗？
3. 用户明确确认同意了吗？

**三问不全 YES，不执行。**

## 破坏性操作清单
- 删除文件/目录
- 清空数据库/集合
- 覆盖/重写数据文件
- 重置配置
- 停止/重启服务（非用户要求时）
- 任何不可逆的修改

## 例外
除非有明确的紧急情况（如安全漏洞正在被利用），否则不适用任何例外。
